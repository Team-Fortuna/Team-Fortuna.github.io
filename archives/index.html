前置条件

前提：用户进程创建socket，然后使用recvfrom()根据用户传入的fd找到socket对象，遍历接收队列接收数据查看是否有数据。如果没有收到数据或者没有收到足够的数据就启用sk_wait_data阻塞当前进程，sk_wait_data将当前进程(current)关联到所定义的等待队列里，然后将进程状态设为interruptible(也就是阻塞)，调用schedule_timeout让出cpu。自此只要内核收完数据产生就绪事件的时候，就可以通过查找socket上的等待项，进而找到回调函数和等待该socket就绪事件的进程了。切换上下文，等待数据到达网卡。

CPI（Cycle Per Instruction ）：每指令周期数。
IPC（ Instruction Per Cycle）： 每周期指令数。
CPI = 1 / IPC
CPI 小于 1，程序通常是 CPU 密集型的；
CPI 大于 1，程序通常是 IO 密集型的;
程序执行时间 = 程序总指令数 x 每CPU时钟周期时间 x 每指令执行所需平均时钟周期数

CPU使用率

CPU的利用率就是指非空闲进程占用时间的比例，即CPU执行非空闲进程的时间 / CPU总的执行时间。
us：运行用户态进程占用时间比例
sy：运行内核态进程占用时间比例
ni改变过优先级的进程占用时间比例
wa等待io完成所消耗时间占比
hi处理硬中断所消耗的时间占比
si处理软中断所消耗的时间占比
st被其它虚拟机抢占的时间占比，你的虚拟 CPU 等待实际 CPU 的时间的百分比。Xen Hypervisor 通过hypercall将vcpu分配给运行在其它虚拟机上的任务的时间占比。由于kvm、vmware和物理机不涉及hypercall所以一般为0

平均负载

Linux load average指标是一个综合了cpu load avg、disk load avg和network load avg得出的系统整体平均负载，主要用于衡量任务对系统的需求。通过对单位时间内系统处于可运行状态和不可中断状态的平均进程数来体现出系统对资源的需求及变化趋势，而不仅仅是CPU需求。
其实Linux load average既不是真正的平均值，而是指数衰减平均数，时间也不是真的1、5、15分钟。1、5、15min 是在等式中使用到的常量，其计算了 5s 平均的 expoentially-damped moving sums 。
更有针对性的指标：

  • ​每个 CPU 利用率​：例如，使用mpstat -P ALL 1
  • ​每个进程的 CPU 利用率​：例如top、pidstat 1等。
  • ​每线程运行队列（调度程序）延迟​：例如，在 /proc/PID/schedstats、delaystats、perf sched
  • ​CPU 运行队列延迟​：例如，在 /proc/schedstat、perf sched中，我的runqlat ​bcc工具。
  • ​CPU 运行队列长度​：例如，使用vmstat 1和“r”列，或我的 runqlen bcc 工具。

前两个是利用率指标，后三个是饱和度指标。利用率指标可用于表征工作负载，而饱和度指标可用于识别性能问题。最好的 CPU 饱和度指标是运行队列（或调度程序）延迟的度量：任务/线程处于可运行状态但必须等待轮到它的时间。这些允许您计算性能问题的严重程度，例如，线程花在调度程序延迟上的时间百分比。相反，测量运行队列长度可能表明存在问题，但更难估计大小。

由于不是所有程序都能利用上cpu百分百的性能，所以usage和load存在一些差别。

平均负载是指单位时间内，系统处于可运行状态和不可中断状态的平均进程数，也就是平均活跃进程数.
